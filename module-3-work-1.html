Collaborative Discussion: UML Diagrams

Assignment brief

My first piece of work for the Secure Software Development module was to create a UML diagram outlining one of the coding weaknesses identified by OWASP. The diagram had to be accompanied by a 200 word description of why the UML model I used is the most appropriate choice. I decided to detail SQL injection using a sequence diagram. My submission is below.

Write up

OWASP have identified SQL injection as a critical vulnerability which can be exploited online. It describes the process of a inserting unwanted SQL commands directly into a database without it being meditated and interpreted by the application server first (Nystrom, 2007). An application or site which does not protect itself from SQL injection can leave itself open to the risk of unwanted data being extracted or entered into the system, or even the deletion of whole databases. There are a number of defences which can be put in place, but the example shown in the sequence diagram is the use of input validation. At its most basic, input validation will check if the input string contains any illegal characters which can be used to alter a query, and only construct the query if it passes the test (Galluccio, 2020). Doing this in a non-SQL based framework, such as Java, will add an additional layer of protection to stop any unwanted input from reaching the database. The sequence diagram presents this very well by clearly differentiating the different actions taken by the individual systems. This makes it very clear that the validation is being done by the application server, rather than the database. It also clearly demonstrates the timeline of each action occurring.

Galluccio, E., Caselli, E., & Lornbari, G. (2020) SQL Injection Strategies. Packt Publishing.

Nystrom, M. (2007) SQL Injection Defences. O'Reilly Media, Inc.
